# Отчёт по работе с Git и Markdown-документацией в проекте «RAG Mobile App»

## Общая информация

В ходе разработки мобильного приложения для взаимодействия с RAG-сервисом использовалась система контроля версий Git и платформа GitHub для хранения исходного кода, отслеживания изменений и организации совместной работы над проектом.

Особое внимание уделялось ведению проектной документации в формате Markdown (`.md`), который является стандартом оформления документации в репозиториях GitHub. Markdown использовался для создания README-файлов, технических отчётов, инструкций по развёртыванию, описания функциональности приложения и фиксации результатов разработки.

# Работа с Git

## Использование системы контроля версий

Для управления разработкой применялась распределённая система контроля версий Git.

Основные задачи Git в проекте:

* хранение истории изменений;
* совместная работа участников команды;
* отслеживание версий исходного кода;
* возможность возврата к предыдущим версиям;
* документирование этапов разработки через систему коммитов;
* синхронизация локальной и удалённой версий проекта.

Каждый коммит представляет собой отдельный снимок состояния проекта и содержит описание внесённых изменений.

---

## Основные Git-команды

В процессе работы использовались следующие команды:

```bash
git clone
git add .
git commit -m "Описание изменений"
git push origin main
git pull origin main
git status
git log
```

### Типичный рабочий цикл

```bash
git pull origin main

git add .

git commit -m "feat: add chat interface"

git push origin main
```

Данный цикл выполнялся после завершения каждого этапа разработки или реализации новой функциональности.

---

## Организация коммитов

Для удобства отслеживания истории проекта каждый коммит сопровождался кратким и информативным описанием изменений.

Примеры сообщений:

* Added login screen
* Updated README
* Fixed API connection
* Added chat functionality
* Updated project documentation

Такой подход обеспечивал понятность истории разработки и позволял быстро находить нужные изменения.

---

## Использование соглашения Conventional Commits

Для поддержания единого стиля ведения истории изменений команда придерживалась общепринятого соглашения Git-сообщества **Conventional Commits**.

Данный стандарт позволяет быстро определить назначение коммита, облегчает сопровождение проекта и повышает читаемость истории изменений.

Каждое сообщение коммита начиналось со специального префикса, обозначающего тип внесённых изменений.

### Используемые префиксы

| Префикс     | Назначение                                   |
| ----------- | -------------------------------------------- |
| `feat:`     | Добавление новой функциональности            |
| `fix:`      | Исправление ошибок                           |
| `docs:`     | Изменение документации                       |
| `chore:`    | Технические изменения и обслуживание проекта |
| `refactor:` | Рефакторинг без изменения функциональности   |
| `test:`     | Добавление или обновление тестов             |
| `style:`    | Изменения форматирования и оформления кода   |

### Примеры коммитов

```bash
feat: add authentication screen

feat: implement chat page

fix: resolve navigation bug

docs: update project documentation

chore: update dependencies

refactor: simplify API service

test: add login screen tests
```

Использование Conventional Commits обеспечило единообразное оформление истории проекта и соответствие современным практикам разработки программного обеспечения.

# Работа с Markdown

## Назначение Markdown

Markdown представляет собой лёгкий язык разметки, предназначенный для оформления технической документации.

В проекте Markdown использовался для:

* README.md;
* отчётов по этапам разработки;
* описания архитектуры приложения;
* документации по запуску проекта;
* ведения журналов изменений;
* оформления инструкций для разработчиков.

Основным преимуществом Markdown является простота написания и автоматическое отображение документации непосредственно на GitHub.
